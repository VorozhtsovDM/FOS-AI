# Проектная работа

**Оценочное средство:** ОС-4 — защита итогового проекта  
**Форма:** индивидуальный аналитический проект по кейсу индустриального партнёра + защита  
**Тип оценивания:** суммирующее

## Назначение

Проектная работа закрепляет навыки решения прикладной задачи машинного обучения в бизнес-контексте индустриального партнёра (Сбер, Ozon, РТК ИТ): от постановки и EDA до сравнения с baseline, интерпретации метрик и предложения по внедрению.

Тема проекта выбирается из каталога [кейсов индустриальных партнёров](../resources/tasks/README.md#кейсы-индустриальных-партнёров) ([PDF](../resources/tasks/cases.pdf)).

|---|---|

Роли КРМ: **Data Analyst**, **ML Engineer**.

## Условия выполнения

1. Студент выбирает **один** кейс из каталога (Р-1…Р-3, Кл-1…Кл-3 или Кз-1…Кз-3).
2. Работа выполняется на **публичном датасете-аналоге**, указанном в описании кейса (реальные данные партнёров закрыты).
3. Обязательны: EDA, baseline из описания кейса, ≥2 подхода (модели / алгоритма / настройки), сравнение по метрикам кейса, бизнес-интерпретация и сценарий внедрения.
4. Для временных задач — корректный split по времени; для несбалансированной классификации — учёт дисбаланса и метрики не только accuracy.
5. Срок сдачи отчёта и защиты — по календарю дисциплины (занятие 2.7).

## Формат сдачи

### 1. Письменный отчёт (до защиты)

Структура — по [шаблону в каталоге кейсов](../resources/tasks/README.md#шаблон-отчёта-студента):

1. Партнёр, ID кейса, бизнес-проблема.  
2. Заинтересованные роли и ожидаемое внедрение.  
3. Данные: источник-аналог, объём, предобработка.  
4. Baseline и выбранные модели / алгоритм.  
5. Метрики и сравнение с baseline.  
6. Бизнес-интерпретация и предложение по внедрению.  
7. Выводы.

Формат файла: Markdown или PDF; воспроизводимый код — Jupyter Notebook / скрипт с указанием версий библиотек.

### 2. Защита перед комиссией

- Презентация (5–7 слайдов) и демонстрация решения (ноутбук / ключевые результаты).  
- Регламент: ~7–10 минут доклад + вопросы.  
- На защите студент кратко отвечает: постановка → данные → baseline vs ML → метрики → внедрение.

## Критерии и шкала оценивания

Оценка выставляется по 10-балльной шкале ФОС (уровни КРМ). Рубрикатор согласовывается с критериями каталога кейсов.

| Критерий | Вес | 8–10 | 6–7 | 4–5 | 0–3 |
|---|---|---|---|---|---|
| Постановка и связь с бизнесом | 15% | Чёткие Y/объект/метрики, роли, сценарий внедрения | Постановка ясна, внедрение схематично | Постановка неполная | Нет связи с кейсом |
| Данные и EDA | 15% | Корректная предобработка, осмысленный EDA | EDA есть, мелкие пробелы | Поверхностный EDA | EDA отсутствует / данные использованы некорректно |
| Baseline и модели | 20% | Baseline + ≥2 подхода, обоснованный выбор | Baseline и сравнение есть | Один подход без сильного baseline | Нет сравнения |
| Метрики и выводы | 20% | Метрики кейса, сравнение с baseline, выводы согласованы с цифрами | Метрики посчитаны, выводы в целом верны | Метрики частичны / не те | Выводы без опоры на результаты |
| Внедрение | 15% | Пороги/действия, роли, план пилота | Есть сценарий использования | Общие фразы | Только «обучил модель» |
| Защита и оформление | 15% | Понятный доклад, ответы на вопросы, воспроизводимость | Доклад уверенный, ответы частичные | Слабая презентация | Не готов к защите |

**Порог зачёта по проекту:** не ниже **4** баллов.  
**Ориентир «сдано»:** корректный эксперимент **и** осмысленный бизнес-вывод с планом внедрения, а не только таблица метрик.

Итоговый балл по ОС-4 учитывается в оценке дисциплины согласно [паспорту ФОС](../docs/pasport.md) (проект / защита перед комиссией).

## Правила использования внешних ресурсов и генеративного ИИ

**Допускается:**
- публичные датасеты-аналоги и документация библиотек;
- LLM для справки по API, черновика текста, подсказок по отладке — при условии, что студент понимает и может объяснить каждое решение на защите;
- указание в отчёте использованных внешних материалов и инструментов (включая LLM).

**Не допускается:**
- сдача чужого кода/отчёта без самостоятельной доработки и понимания;
- подмена результатов эксперимента сгенерированными «метриками» без реального прогона;
- использование закрытых данных партнёров;
- сокрытие факта существенной помощи LLM при вопросах на защите.

Нарушение правил ведёт к оценке **0–1** по соответствующим критериям и/или направлению на пересдачу по решению преподавателя.
